### Side-eye bug — definitive fix (5th attempt, succeeded)

The face/head were persistently looking hard to the right regardless of POI position or mobility state. Solved in three layers:

**Fix 1 — atan2 from world origin vs. robot position (`trackWorldPoint`, line ~5612):**
```js
// BEFORE (wrong when robot not at origin):
const targetYawDeg = Math.atan2(targetPos.x, targetPos.z) * 180 / Math.PI;
// AFTER:
const targetYawDeg = Math.atan2(targetPos.x - drivePos.x, targetPos.z - drivePos.z) * 180 / Math.PI;
```

**Fix 2 — Two-tier sanity check in `preUpdate` (line ~5690):**
- `headRotationMax = 25°`; head saturates at 33°; old check only fired at 50° — gap of 17° where head was pinned but not released
- When MOVING: release POI at `headRotationMax + 10°` (35°), set `behaviorTimer = -1.0` (1s cooldown prevents immediate re-pick via `selectAttentionPOI`)
- When STATIONARY: keep `poiIdx` set (body-turning code needs it), suppress `selectAttentionPOI` by setting `behaviorTimer = -0.5` each frame, re-arm `attentionHoldTimer` to prevent expiry while body turns

**Fix 3 — driveYaw normalization — THE ACTUAL ROOT CAUSE (`trackWorldPoint`, lines ~5593, 5612):**
Recording showed `body3DYaw: -2856.57`. `driveYaw` accumulates forever without normalization. Raw subtraction gave `relYawDeg ≈ 2889°` → head/eyes permanently pinned to max right.
```js
// Normalize body3DYaw on assignment:
s.body3DYaw = ((driveYaw * 180 / Math.PI) % 360 + 540) % 360 - 180;
// Normalize relYawDeg after subtraction:
const relYawDeg = ((targetYawDeg - s.body3DYaw) % 360 + 540) % 360 - 180;
```
The sanity check in `preUpdate` already used `% 360` normalization (correct), but `trackWorldPoint` did raw subtraction (wrong). Confirmed via verbose recording log.

**All three fixes together resolved the issue. User confirmed working.**

## 4. Key Decisions & Rationale

- **Use recording logs first** — five sessions of blind code inspection failed to find the normalization bug; one verbose recording found it immediately. Always check the recording before editing.
- **Two-tier sanity check** — moving vs. stationary distinction is important. Stationary needs `poiIdx` kept so the hold-mode body-turning code (`preUpdate` line ~5795) knows where to spin. Releasing `poiIdx` would break that feedback loop.
- **`behaviorTimer` is the gate, not `attentionHoldTimer`** — `selectAttentionPOI()` is called when `behaviorTimer >= 0`, ignoring `attentionHoldTimer`. Setting only `attentionHoldTimer` for cooldowns doesn't work; must set `behaviorTimer` to a negative value.
- **Handover skill** — no longer writes `HANDOVER.md`; only timestamped files in `handovers/`. More organized, nothing gets overwritten.

## 5. Pitfalls & Lessons Learned

- **`driveYaw` NEVER normalizes** — it accumulates forever. Any math using `driveYaw` (or `s.body3DYaw` derived from it) for angle comparison MUST use `% 360` normalization. Formula: `((angle) % 360 + 540) % 360 - 180`. The preUpdate sanity check had this right; `trackWorldPoint` did not.
- **Sanity check passes, tracking still wrong** — the sanity check in `preUpdate` was correct (normalized), so it allowed `trackWorldPoint` to run, which then computed a wrong (huge) angle. Don't assume a passing sanity check means downstream math is OK.
- **`selectAttentionPOI` overrides `attentionHoldTimer` immediately** — calling it sets both `poiIdx` and `attentionHoldTimer` to new values, wiping any cooldown set externally. Use `behaviorTimer` (negative = paused) to prevent re-picks.
- **`trackWorldPoint` resets `behaviorTimer = -0.5` every frame** — this is why saccades are rare when a POI is tracked. It's intentional (suppresses saccade planner during tracking) but has the side effect of near-zero saccades in practice.
- **All prior invariants still apply** — see CLAUDE.md for the full list.
